Overview
This tool is the 'CollaboRate Work Breakdown Wizard'. It is used by online workers to do their work.

This tool is part of the open source project 'CollaboRate'. One of the main goals of this project is to create opportunities for anyone to earn online by doing small, easy, fun, and socially useful intellectual tasks. The work done is used to develop, maintain and test software apps.
Benefits
This tool is part of a new way of software development, that has the following benefits:
  • Allows people to earn money by doing online work, in an enjoyable, flexible and socially useful way.
  • Provides people ample time to learn the skills and industry knowledge needed to do the work.
  • It provides a very accurate way to measure people's work. This in turn allows people to be paid in a very fair manner.
  • Better quality code is created that is very simple and very well documented.
  • This way of software development allows a very large number of people to work on the same project at the same time. So it is very fast when the number of people working on the project are very large in number.
  • It reduces financial risks for those who need software to be developed. This is done by sharing the risk in a fair and small way with all people, including those working on the project. The risk per person becomes smaller, when there are more workers on the project.


Skills Needed by the Online Workers
As a 'crowd member' (one of the online workers) you must have the following skills:
  • Able to read and write English clearly and correctly. As of now, this tool is limited to the English language.
  • Willing to learn a small amount of any additional publicly available information needed to work on the small task. You will have ample time to learn and complete the task.
  • A basic understanding of how a software app works and some related terms, at the level taught to 8-year-olds in school. This is specified as follows:
    • An app is a set of instructions written by people, that tell a computer how to do a particular task. Examples of apps are Microsoft Word, Excel, Powerpoint, apps installed on smartphones, etc.
    • An app accepts various types of inputs (information). Inputs may be provided by:
      • a person (user of the app) via electronic machines called 'input devices'. Examples of such input devices are keyboard, mouse, touchscreen etc.
      • an alarm set on the computer's clock
      • some other app
      • other input devices that can automatically provide input whether the user is present or not. For example, microphone, camera etc.
    • The app then 'processes' these inputs. This means that it does a list of things called 'steps' on these inputs.
    • Some of these steps may create part of the results that we expect to see or hear from the computer (the output).
    • Other steps may update the data stored by the app in its 'database' ('db', permanent memory of the computer whose data is not lost even if the computer is switched off and back on again).
    • Yet some other steps may cause a message to be sent to another computer.
The Process

  • Open source projects or commercial groups called 'Crowdsourcing vendors' advertise online work opportunities to the public. We term these potential workers as 'crowd members'.
  • The crowd members register on the website of the crowdsourding vendor and then submit a request for work.
  • The crowdsourcing vendor assigns some text to the crowd member that does not exceed 2000 characters. This text is a description of a part of the software app being developed . It is also known as a task or microtask. Once this text is assigned, the crowd member will receive a link to this text via email, or in the appropriate section of the vendor's website that is visible once the member has logged in.
  • The crowd member will click on the link. This tool will then open up in the web browser of the crowd member, with the assigned description also loaded.
  • This tool asks the member a series of questions about the description that was given to the member. For most of these questions, the member has to choose the right answer from a list of possible answers. Based on the previous choice made by the member, the tool will decide the next question to ask.
  • The tool captures the answers provided by the member and submits them to the website of the crowdsourcing vendor.
  • Depending on the agreements between the crowdsourcing vendor and the crowd members, the crowd members will get paid based on the number of micro-tasks done correctly and in time.
  • The crowdsourcing vendor may also ask for a tiny amount of money as a security deposit from the crowd member. If delays or errors caused by the crowd member cause a loss that is more than what can be deducted from the payment amount per microtask, a part of this deposit may be used up.
How It Works
Note: It is not necessary for the crowd members to understand the below in order for them to do their work.
  • By answering the series of questions asked by the tool, you break down (divide) the partial description of the app, into one or two more such partial descriptions of the app. Let's call the broken-down descriptions 'sub-descriptions'.
  • Assume that the crowd member answers the questions asked by this tool correctly and as instructed. In that case, these sub-descriptions along with the answers to the objective-type questions, will usually contain neither more nor less scope and information than the original description.
  • In some cases, the descriptions will become more detailed. In some other cases, a small amount of new information might need to be introduced, based on publicly available information about the industry area of the application that is being developed.
  • Whether or not more elaboration of the problem has happened or more information has been added, each sub-description must contain all the needed information, such that it is not needed to refer to the parent description in order to understand the sub-description. Each sub-description must also remove that part of the parent description that is not of relevance to it.
  • Due to the above rules for the sub-descriptions, these sub-descriptions can be further provided to any other crowd members to break down, and so on, until it is not possible to break the sub-descriptions down any further.
  • The answers to some of the objective-type questions determine one of the processing steps done by the app. In this way, crowd members help to create all the processing steps (source code) of the app!
  • The 'binary' work breakdown way of working allows us to very accurately measure work done by the crowd members. The current assumption is that the average effort needed to break down a task is almost the same, especially given that there is a limit imposed on the maximum number of characters of task's text. So we can simply measure of the contributions of the crowd members by counting the number of microtasks that each crowd member has broken down.
    Even if it is found that different tasks need different amounts of effort later on, we can refine this system of work measurement slightly. The basic system of work measurement seems likely to remain the same.
  • Creating multiple subtasks helps to allow multiple crowd members to work on the same project in parallel, so that the project gets over quickly when a large number of crowd members are available.
  • Creating multiple subtasks provides security for those projects that need it. Crowdsourcing vendors can ensure that each crowd member gets to work on only a small part of the project. This way, no single crowd member knows the entire project design. If the project requirements are themselves confidential, then the crowdsourcing vendor can itself do the work breakdown up to some level and then crowdsource the rest of the work.
  • To ensure the quality of the work being done by the crowd and to troubleshoot (find the cause of errors), a number of techniques can be used by the crowdsourcing vendor. These must include:
    • manual acceptance testing - performed by the crowdsourcing vendor and/or the end customer
    • unit testing - used to troubleshoot that which work breakdown done by the crowd, is incorrect.
    • automation of regression testing - used to reduce the manual testing efforts and errors. Both acceptance testing and unit testing can be automated.
    In additional to the above mandatory testing, crowdsourcing vendors may also ensure quality by the asking the crowd members to review each others' work (at the cost of some small loss of security). Finally the crowdsourcing vendors may use a variety of Natural Langauge Processing and other AI techniques to check the the crowd members have done the work correctly. Some such validations may be offered for free as a part of this tool.
