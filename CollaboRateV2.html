Welcome to the the CollaboRate work breakdown wizard v2.0. This tool is part of the Open Source CollaboRate project.
One of the main goals of this project is to create opportunities to earn online by doing small, easy, fun, and socially useful intellectual tasks.
This intellectual work done is used to develop, maintain and test software apps.
The information below should be enough to get you started with the tool and process.
For more information, visit the project website.

Benefits

This tool is part of a new way of software development, that has the following benefits:
  • It allows for greater employment opportunities in the field of software development. This is because it allows people to develop software even if they do not have the skills and knowledge usually associated with software development jobs.
  • Allows people to earn good amounts of income online, in an enjoyable, flexible and socially useful way.
  • It provides a very accurate way to decide that who has contributed how much to the project. This in turn allows the people who worked on the project to be paid in a very fair manner.
  • Much better quality code is created.
  • This way of software development allows a very large number of people to work on the same project at the same time. So it is very fast when the number of people working on the project are very large in number.
  • It reduces financial risks for those who need software to be developed. This is done by sharing the risk in a fair and small way with all stakeholders. The risk per person becomes smaller when the number of people working on the project are very large in number.


As a crowd member, you must have the following skills:
  • Able to read and write English clearly and correctly. As of now, this tool is limited to the English language.
  • Willing to learn a small amount of any additional publicly available information needed to work on the small task.
  • A basic understanding of how a software app works and some related terms, at the level taught to 8-year-olds in school. This can be summarized as follows:
    • An app is set of instructions written by people, that tell the computer how to do a particular task. Examples of apps are Microsoft Word, Excel, Powerpoint, apps installed on smartphones, etc.
    • An app accepts various inputs (information) for example: text, video, pictures or sound. Inputs may be provided by:
      • a person (user of the app)
      • an alarm set on the computer's internal clock
      • some other app
    • The app then does processing (a list of logic and/or maths steps) on these inputs.
    • Some processing steps create part of the output.
    • Some processing steps update to the data stored by the app. The app may store data on hard disks or in case of mobile apps, on the 'permanent memory' of the smartphone.
    • Like the input to an app, the output produced by an app is also information that may be in various forms.
    • A particular type of input-process-output is called an 'interaction'.
The Process

  • Open source projects or commercial groups called 'Crowdsourcing vendors' advertize online work opportunities to the public. We term these potential workers as 'crowd members'.
  • The crowd members respond to these ads by requesting for work.
  • Descriptions of parts of the software app being developed will be shared with the crowd members, along with a link to this tool. The description given to the user is also known as a task or microtask.
  • Users will click on the link provided and this tool will open up in their web browsers.
  • This tool asks the user a series of questions about the description that was given to the user. For most of these questions, the user has to choose the right answer from a list of possible answers. Based on your previous choice, the tool will decide the next question to ask.
  • The tool captures the answers provided by the user and submits them online to the website of the crowdsourcing vendor.
  • Depending on the agreements between the crowdsourcing vendor and the crowd members, the crowd members will get paid based on the number of micro-tasks done correctly and in time.
How It Works
  • By answering the series of questions asked by the tool, you break down (divide) the partial description of the app, into one or two more such partial descriptions of the app. Let's call the broken-down descriptions 'sub-descriptions'.
  • Assume that the crowd member answers the questions asked by this tool correctly and as instructed. In that case, these sub-descriptions along with the answers to the objective-type questions, will contain neither any more or less scope and information than the original description.
    In some cases the descriptions might need to be slighly more elaborated. Or a small amount of new information might need to be introduced, based on publicly available information about the industry area of the application that is being developed.
    Whether or not more elaboration of the problem has happened or more information has been added, each sub-description must contain all the needed information, such that it is not needed to refer to the parent description in order to understand the sub-description. Each sub-description must also remove that part of the parent description that is not of relevance to it.
    The sub-descriptions can then be further provided to crowd members to break down, and so on.
  • The answers to some of the objective-type questions determine one of the processing steps done by the app. In this way, crowd members help to create all the processing steps (source code) of the app!
  • The 'binary' work breakdown way of working allows us to very accurately measure work done by the crowd members. The current assumption is that the average effort needed to break down a task is almost the same, especially given that there is a limit imposed on the maximum number of characters of task's text. So we can simply measure of the contributions of the crowd members by counting the number of microtasks that each crowd member has broken down.
    Even if it is found that different tasks need different amounts of effort later on, we can refine this system of work measurement slightly. The basic system of work measurement seems likely to remain the same.
  • Creating multiple subtasks also helps to allow multiple crowd members to work on the same project in parallel, so that the project gets over quickly when a large number of crowd members are available.
Instructions for Crowdsourcing Vendors
This tool is part of the open source (MIT licensed) CollaboRate project. This tool is open sourced to serve as a standard interface for Crowd Members to use to perform the binary work breakdown, irrespective of the crowdsourcing vendor that hosts this tool.
Once the following are identified:
  • Subtask 1 of the task
  • Subtask 2 of the task
  • Logic that connects the 2 subtask
...you need to create 3 data nodes for each of the above in your db, via calling the appropriate js functions from your copy of this tool. For each of these three nodes, the problem statement that was broken down using this tool is the parent node. You will further validate the input provided by the crowd member either manually or programmatically. You will use the inputs provided by the crowd members to generate the documentation and source code of the app.
